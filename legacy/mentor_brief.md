# 멘토 브리핑

작성 시점: 2026-07-08

## 현재 상황

- 대회: DACON 236694 AI Agent 다음 행동 14클래스 예측 문제임.
- 평가: 14개 클래스별 F1 점수 평균(Macro-F1)임.
- 현재 목표: 현재 본선 컷 기준 `0.787` 도달임.
- 현재 최고 LB(공개 리더보드 점수): `0.7677217115`임.
- 현재 순위: 약 50등권임.
- 컷까지 격차: 약 `+0.0193` 필요함.
- 현재 멘토 전달 제출: `submit.zip`임.
- 전달 제출 구성: Qwen3-0.6B-Base, 3epoch, fold0(첫 번째 데이터 분할) 단일 모델, max_len 512, au prior 보정 포함임.
- 전달 제출 방식: `requirements.txt`로 transformers 4.51.3 계열 설치 후 Qwen3 추론 실행함.
- 최고 제출 런타임: `9분 31초`임.
- 전달 제출 zip 용량: 약 `839MB`임.

## 용어

- LB: 공개 리더보드 점수. 데이콘 서버에 제출해서 숨겨진 테스트 데이터로 채점된 점수임.
- CV: 내부 검증 점수. train 데이터를 나눠서 로컬에서 예측 성능을 추정한 점수임.
- OOF: 5분할 전체 내부 검증 점수. 각 fold의 검증 예측을 모아 전체 train 기준으로 계산한 점수임.
- fold0: 첫 번째 데이터 분할임.
- max_len: 모델에 넣는 최대 입력 토큰 길이임.
- au prior: au 구간에서 자주 나오는 action 분포를 반영한 사전확률 보정임.

## 대회 제약

- 코드 제출 방식임. `script.py`와 모델을 zip으로 제출함.
- 공식 페이지 기준 추론 코드 실행 시간은 10분 이하임.
- 공식 페이지 기준 패키지 설치 시간은 10분 이하임.
- 공식 페이지 기준 제출 파일 용량 제한은 1GB 이하임.
- 공식 페이지 기준 실행 환경은 오프라인임. 단 패키지 설치 외 인터넷 연결 불가 조건임.
- 공식 페이지 기준 평가 환경은 T4 GPU 16GB VRAM, 3 vCPU, 12GB RAM임.
- 현재 제출 한도는 zip 파일 기준으로 보고 있음. 비압축 원본 용량은 참고값임.

## 지금까지 결과

| 항목 | 결과 |
|---|---:|
| Qwen3-0.6B fold0 3ep CV | `0.7679242649` |
| Qwen3-0.6B fold0 3ep LB | `0.7677217115` |
| Qwen3-0.6B 5-fold OOF | `0.7701255471` |
| Qwen3 5-fold weight soup 제출 LB | `0.7602664582` |
| Qwen3 5-fold weight soup 런타임 | `9분 35초` |
| Qwen3 x XLM-R OOF 블렌드 | 약 `0.7737` |
| Qwen2.5-Coder-0.5B fold0 | `0.7571` |
| Qwen3-0.6B 4ep fold0 | `0.7616` |

- Qwen3 fold0 LB와 CV 차이가 거의 없음. 현재 CV 방식은 신뢰도가 높아 보임.
- Qwen3 5-fold OOF는 단일보다 높지만, 5개 모델을 그대로 넣을 수 없음.
- weight soup는 5개 fold 모델의 가중치를 평균내서 단일 모델에 담으려는 시도였으나 LB가 `0.7603`으로 낮아짐.
- Qwen3 x XLM-R 블렌드는 OOF에서 `0.7737`로 개선됨. 다만 현재 구성에서는 Qwen3 모델 약 905MB + XLM-R 약 200MB + 라이브러리 문제로 1GB/10분 제약에 걸림.
- Qwen2.5-Coder는 fold0 `0.7571`, Qwen3는 fold0 `0.7679`라서 Qwen3가 더 좋아 Qwen2.5-Coder는 우선순위에서 빠짐.
- Qwen3 4epoch는 3epoch보다 낮아짐. 현재는 3epoch가 기준임.

## 서버 라이브러리 상태

- 평가 서버 기본 transformers는 Qwen3를 직접 지원하지 않는 버전으로 확인됨.
- 현재 제출 `submit.zip`은 `requirements.txt`로 `transformers==4.51.3`, `tokenizers==0.21.0`, `huggingface_hub==0.30.0`을 설치하는 방식임.
- 이 `requirements.txt` 설치형은 서버 정상 작동 확인됨.
- 따라서 Qwen3 제출은 별도 라이브러리 폴더 동봉 없이 가능함.

## 현재 막힌 점

- turn0-1 구간이 거의 개선되지 않음. Qwen2.5에서 Qwen3로 바꿔도 `0.424 -> 0.425` 수준임.
- 5-fold OOF는 `0.7701`로 단일보다 높지만, 5개 모델을 그대로 제출하면 1GB/10분 제약에 걸림.
- weight soup는 5-fold 이득을 단일 모델에 담으려는 시도였으나 LB `0.7603`으로 기존 단일 `0.7677`보다 낮아짐.
- Qwen3 x XLM-R 블렌드는 OOF `0.7737`로 개선되지만, 현재 형태로는 모델 크기와 런타임 때문에 제출이 어려움.
- 후처리 bias, label adjustment, threshold, 간단 동사 rule은 OOF/cross-fit에서 개선이 없었음.
- Qwen3 4epoch는 `0.7616`으로 3epoch `0.7679`보다 낮아짐. 단순히 epoch를 늘리는 방향은 좋지 않았음.
- 1.5B 이상 모델이나 다중 모델 전략은 T4 16GB, zip 1GB, 추론 10분 조건 때문에 바로 쓰기 어려움.
- 현재 최고 제출도 `9분 31초 / 10분`이라 런타임 여유가 약 30초뿐임. 모델 추가, 입력 길이 증가, 복잡한 후처리가 모두 위험함.

## 가장 큰 미흡사항

- turn0-1 성능이 낮음.
- Qwen2.5에서 Qwen3로 바꿔도 turn0-1 macro-F1은 `0.424 -> 0.425` 수준으로 거의 개선되지 않음.
- 반면 turn2+ 구간은 Qwen3에서 약 `0.767` 수준까지 나옴.
- turn0-1은 히스토리가 짧아서 현재 직렬화/모델 구조만으로는 정보가 부족한 구간으로 보임.
- 그래도 전체 점수 상승 폭 관점에서는 이 구간이 가장 큰 미개척 영역일 가능성이 있음.
- 현재 turn0-1 전용 학습, oversampling, 별도 head, 별도 모델, 메타/프롬프트 패턴 설계는 충분히 깊게 하지 못한 상태임.

## 멘토님께 물어볼 질문

1. turn0-1 성능을 올리려면 모델 크기보다 데이터 표현/학습 구조를 바꾸는 게 맞는지 확인 필요함. 후보는 turn0-1 전용 head, turn bucket별 loss weighting, turn0-1 oversampling, current_prompt 중심 직렬화, session_meta 강화, 클래스별 action prior 설계임.
2. Qwen3 x XLM-R 블렌드가 OOF에서 `0.7737`까지 오르는데 1GB/10분 제약에 걸림. 모델 프루닝, int8/동적 양자화, max_len 384 축소 중 어떤 조합이 가장 현실적인지 판단 필요함.
3. Qwen3.5 계열은 아직 본격 실험하지 않음. 범용 모델이라 우선순위가 밀렸고, 현재는 Qwen3-0.6B가 검증된 기준임. 다만 Qwen3에서 서버 transformers 교체 방식이 성공했으므로 조건 맞는 새 모델이면 서버 버전 교체 자체는 할 만함.
4. 후처리 bias, label adjustment, 간단 동사 rule은 OOF/cross-fit에서 개선이 없었음. 추가 후처리보다 학습 구조를 바꾸는 쪽이 맞는지 확인 필요함.

## 바로 가능한 다음 작업

- 다음 우선순위는 turn0-1 전용 학습 실험과 max_len 384 블렌드 가능성 확인임.
- GPU를 오래 쓰는 대형 모델 추가 학습보다, 현재 병목인 turn0-1/용량/런타임 제약을 먼저 푸는 게 합리적임.
